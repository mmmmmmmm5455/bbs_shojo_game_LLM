# 图形版可玩性与性能问题报告（Pygame）

日期：2026-04-08  
目标：让 `main_advanced_v2.py` 像正常视觉小说/聊天游戏一样可输入、可互动，并消除严重卡顿。

---

## 一、用户可复现问题（现状）

### 1) “不知道哪里输入 / 无法正常游玩”
- **现象**：图形版窗口没有明显的文本输入框或聊天区域，玩家无法像“和小误聊天”那样输入并推进体验。
- **原因（设计层）**：当前 `main_advanced_v2.py` 主要是按键触发的演示集合（论坛、打字挑战、帮助层等），缺少“主玩法循环”（聊天输入 → 生成回复 → 历史记录 → 可持续推进）。

### 2) 严重卡顿、输入延迟、CPU 占用高
你反馈的现象：
- 低帧率（鼠标移动/画面刷新都卡）
- 输入延迟
- 偶发卡住几秒
- CPU 很高

---

## 二、关键性能证据（热点定位）

### CRT 后处理是主要瓶颈（P0）
在 `bbs-shojo-game/src/crt_manager.py` 中，`CRTManager.apply_crt_effect()` 对 1024×768 的整帧进行逐像素处理（PixelArray + 双层 for 循环 + 噪点逐点 set_at）。  
这类实现会导致 **每帧数百万级操作**，并触发 Python 解释器层面的巨量开销。

**本地测量（1024×768，aging_level=75）：**
- 单次调用耗时约 **2.25 秒/帧**
- 换算帧率约 **0.4 FPS**

这与“低帧率 + 输入延迟 + CPU 爆高”的用户体验完全一致。

### 修复后测量（优化版 CRT）
在 `crt_manager.py` 采用 **降采样 + 分层缓存叠加** 的低成本实现后，1024×768 下单次调用大幅下降：\n
- `quality=low`：约 **0.015 秒/帧**（~66 FPS 等价）\n
- `quality=medium`：约 **0.044 秒/帧**（~22 FPS 等价）\n
- `quality=high`：约 **0.313 秒/帧**（~3 FPS 等价，仅供调试/展示）\n

默认建议使用 `low`（保证可玩性），必要时才提高质量。

---

## 三、修复方案概述（按优先级）

### P0：把 CRT/VH 后处理改成低成本实现
- **降采样**：对屏幕缩小到 1/2 或 1/3 再处理，随后 scale 回去。
- **分层缓存**：扫描线、噪点、烧屏等用预生成 `Surface` 叠加，每隔 N 帧更新一次。
- **质量档位**：`low/medium/high`，默认 `low`（保证可玩）。
- **必要时降级**：后处理超过时间预算时自动降低质量。

### P1：补全“聊天输入框 + 历史记录”主玩法
在图形版窗口内提供：
- 底部输入框、光标闪烁、Enter 发送、Backspace 删除
- 历史消息区域（自动换行与滚动）
- 与终端版共享同一逻辑（`story_mode/bbs_engine.py`），让玩法一致

### 本次已实现（2026-04-08）
- 图形版 `main_advanced_v2.py` 已加入 **聊天输入框 + 历史记录 + 滚动**，并支持 slash 命令：\n
  - `/help` `/mode` `/sig` `/geo` `/deadlinks` `/memories` `/turing` `/answer` `/theseus`\n
- CRT 后处理已优化为低成本实现（见上文测量）。

## 四、回归验证（完成修复后）
- 启动后立即可输入并发送 20 条消息，不卡顿。
- `H` 帮助中包含“如何输入与发送”。
- CRT/VH 开启时仍可保持可用帧率；若开高质量会提示并可切换性能模式。
